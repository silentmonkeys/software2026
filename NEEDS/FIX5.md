# FIX5 — 功能修复与完善提示词

> **目标工具**: Claude Code  
> **目标模型**: GLM-5.2  
> **项目**: 设备检修知识检索与作业系统（前后端全栈）  
> **参考文档**: `CLAUDE.md`、`NEEDS/` 目录下已有文档、`CHANGE/` 目录  
> **工作原则**: 所有改动必须同时覆盖 `Pc.vue` 和 `Mobile.vue`；保持现有 UI 风格，禁止引入新配色方案或新布局体系；严禁产生任何假数据或 Mock 数据。

---

## 背景与约束

本次修复基于现有前后端工程，前端为 Vue 3 + Vite + TypeScript，后端为 FastAPI + SQLAlchemy + Chroma。修改时：

1. **完全保持现有 UI 风格**：颜色、字体、间距、组件风格均沿用现有设计系统，不引入任何新配色或新布局。
2. **PC 与移动端同步**：凡涉及界面改动，`Pc.vue` 和 `Mobile.vue` 必须同步修改。
3. **无 Mock 数据**：移除所有 `safeCall`/`rawCall` 中的 fallback 假数据，接口失败时展示明确的错误状态组件，不允许用假数据兜底。
4. **角色枚举统一**：前端 `frontline | auditor | admin`，后端 `worker | leader | admin`，所有改动使用统一来源（建议提取到共享常量），确保 `mapBackendRole()` 映射不漏项、不错项。
5. **默认管理员账户**：用户名 `admin`，默认密码 `123456`（原为 `admin123`，需同步修改后端 seed 逻辑），该账户不可被删除。

## 第一部分：账户与权限体系

### 1. 注册界面仅支持注册员工账户

- 注册页面(`/register`)去除角色选择字段，后端注册接口 `POST /api/auth/register` 固定将新注册用户角色设为 `worker`（前端映射 `frontline`）。
- 如果当前注册接口允许传入角色参数，在后端进行强制覆盖，忽略前端传入的 `role` 字段。
- 注册页面说明文字更新为"注册员工账户"。

### 2. 密码管理规则

- **默认密码统一改为 `123456`**：更新后端 `app/main.py` 或 seed 脚本中的默认管理员密码哈希，同时在系统文档/README 中注明。
- **每个用户可修改自己的密码**：在个人中心（`/profile`）提供"修改密码"功能，校验旧密码后允许设置新密码，前后端均实现（`PUT /api/auth/change-password`）。
- **管理员可重置任意账户密码**：管理员账户管理页面中，每个账户条目提供"重置密码"操作，重置后密码恢复为 `123456`，使用 `PUT /api/admin/users/{id}/reset-password` 接口。
- **管理员不能删除默认 admin 账户**：后端在删除账户接口中判断目标账户是否为默认管理员（可通过 `username == "admin"` 或数据库标记字段 `is_default_admin` 识别），若是则返回 `403` 并提示"默认管理员账户不可删除"。

### 3. 管理员账户管理功能

- 管理员拥有独立的账户管理页面（`/admin/users`），可以：
  - 创建新账户（员工/审查员/管理员均可创建）
  - 修改任意账户的用户名、角色、密码（重置为 `123456`）
  - 删除账户（默认管理员除外）
- 创建管理员账户时不需要额外确认步骤，但界面上应有明显的角色提示。
- 表单校验完整，角色字段使用下拉选择。

## 第二部分：知识库与文档管理

### 4. 审查员独立审核界面

审查员（`auditor`）拥有以下专属页面，在导航栏中可见：

- **待审核列表**（`/auditor/review`）：展示员工上传的、待审核的知识条目，支持查看详情、批准/拒绝，批注理由。
- **知识库管理**（`/auditor/knowledge`）：展示当前知识库中已存储的所有文档，支持：
  - 使用 AI 工具辅助（调用现有 `/api/chat/query` 或相关接口）对文档内容进行分析
  - 自由增删文件（调用现有知识库接口）
  - 自由修改知识图谱（调用 `/api/kg` 相关接口）
  - **二次确认机制**：所有修改操作（增、删、改知识图谱）在执行前必须弹出确认对话框，确认后才正式写入，不可撤销的操作（如删除）使用更明显的警告样式对话框。

### 5. 知识上传权限区分

- **员工**（`frontline`）：上传入口仅支持"员工经验分享"，表单包含标题、正文（富文本或 Markdown，支持图片/附件上传），操作界面简洁易用，上传后进入待审核状态。
- **管理员/审查员**：在知识库管理页面可自由添加任意类型的文档，无需审核流程，直接入库。
- 现有上传界面如混用两类逻辑，需按角色拆分或条件渲染。

### 6. 知识库文档导出

- 在知识库文档列表（员工浏览页、审查员管理页）提供"导出"按钮，支持将文档内容导出为 PDF 或 Markdown 文件。
- 后端新增 `GET /api/kb/documents/{id}/export?format=pdf|md` 接口。
- 导出时显示 loading 状态，完成后触发浏览器下载。

### 7. 所有账户可预览知识库文档

- 导航栏增加"知识库"入口，所有登录角色（员工、审查员、管理员）均可访问 `/knowledge/browse`。
- 该页面列出所有已入库的文档，支持搜索、分类过滤，点击可查看文档全文内容。
- 文档内容展示为只读，不提供编辑入口（编辑入口仅在审查员/管理员的管理页面）。

## 第三部分：工单（作业指引）系统

### 8. 工单生命周期与删除逻辑

- **完成后的工单**：员工可在"作业指引"页面自由删除已完成的工单，**不需要填写理由**，系统默认理由为"已完成"，完成记录永久保存。
- **未完成的工单**：删除时必须弹出理由选择对话框，选项为：
  - 需求错误
  - 误触
  - 其他（选择后显示文本输入框，必填）
- **已删除工单的完成记录**可在"个人中心 → 历史工单"中查看，支持将历史工单重新添加到当前作业指引。

### 9. 工单共享与独立进度

- 工单（ticket）属于平台级资源，所有员工创建的工单对其他员工可见。
- 每个工单在每个用户维度有独立的进度状态（`user_ticket_progress` 关联表），互不影响。
- 在员工的"作业指引"主页面：
  - **我的工单**：当前用户已添加的工单，显示自己的进度。
  - **推荐工单**（折叠或次要区域）：其他用户创建但当前用户未添加的工单，作为推荐展示，可一键"添加到我的工单"。
- 添加他人工单时，进度从头开始，与原创建者的进度完全隔离。

### 10. 工单创建时优先推荐已有工单

- 员工发起"创建工单"时，系统先对标题/描述进行相似度搜索（调用 RAG 或关键词匹配），若找到相似度足够高的已有工单，展示推荐列表，用户可直接"添加到我的工单"。
- 只有当已有工单均不满足需求，用户明确选择"创建新工单"后，才进入新建工单流程。

### 11. 工单时间线

每个工单在每个用户维度记录完整时间线，事件节点包括：

| 事件 | 字段 |
|------|------|
| 创建/添加 | `created_at`（若为他人工单则显示"添加时间"） |
| 各步骤完成 | `step_completed_at[]` |
| 工单完成 | `completed_at` |
| 工单删除 | `deleted_at` + `delete_reason` |

- **员工**：只能查看自己的时间线。
- **审查员/管理员**：可查看一个工单被所有用户操作的完整时间线（按用户分组展示）。
- 时间线 UI 使用竖向步骤条展示，保持现有组件风格。

### 12. 个人中心 — 历史工单

- 个人中心（`/profile`）新增"历史工单"标签页。
- 展示当前用户所有已完成或已删除（含理由）的工单记录。
- 每条记录可展开查看自己的时间线，已完成的工单支持"重新添加到作业指引"。

## 第五部分：知识图谱动画优化

### 14. 知识图谱打开动画

现有问题：知识图谱（`/knowledge/graph`）打开时节点全部挤在中心后突然炸开，视觉突兀。

修复方案：

- **初始布局**：节点初始时以最终位置为基准，以轻微的随机偏移量开始，而非全部堆叠在圆心。
- **动画曲线**：使用弹性缓动（`easing: 'elastic.out'` 或等效 CSS 动画）配合较短的持续时间（建议 600–800ms），节点从初始位置平滑过渡到最终位置。
- **分层渐入**：节点和边分两波渐入——节点先出现（透明度 0→1），边稍微延迟（100–200ms 后）绘制，避免一次性渲染导致的视觉噪音。
- **禁止强制仿真初始化**：如使用 D3-force 或 cytoscape，调整 `alpha` 初始值和 `alphaDecay`，使仿真从接近稳定的状态开始，而非从完全随机开始。
- 改动后在 PC 和移动端分别验证动画流畅度。

## 第六部分：清理与工程质量

### 15. 删除所有信创/LoongArch 相关标签

全局搜索并删除以下内容：
- 页面中显示的标签文字，包括但不限于：`LoongArch`、`银河麒麟`、`信创认证`、`LoongArch + 银河麒麟 V11`、`LoongArch64`
- 以上文字对应的标签组件及其背景样式（如带色块的 badge/chip 组件）
- 注意：`CLAUDE.md` 和 `NEEDS/` 等文档文件不在清理范围内，只清理前端渲染的 UI 内容

### 16. 彻底移除 Mock 兜底机制

- 删除 `src/api/request.ts` 中 `safeCall` 和 `rawCall` 的 fallback 参数默认值，改为接口失败时 `throw` 或返回 `null`。
- 删除所有 `src/api/<domain>.ts` 中的 mock fallback 对象（通常是 `const mock... = {...}` 形式的变量）。
- 在各视图组件中，用真实的错误状态 UI（空状态组件、错误提示 banner 等）替代原来的 fallback 数据展示。
- 确保移除后系统在无后端时明确提示"服务不可用"，而非展示假数据。

### 17. 修复角色枚举割裂问题

- 在前端新建 `src/constants/roles.ts`，统一定义前端角色常量和后端角色常量，以及 `mapBackendRole()` 映射函数，作为单一来源（single source of truth）。
- 将所有散落在各文件中对角色字符串的硬编码引用替换为该常量。
- 检查并补全菜单权限 `MENU_ITEMS` + `getVisibleMenuItems(role)` 中所有角色分支，确保 `auditor` 和 `admin` 的菜单项与本次新增页面保持一致。
- 后端同步：确认 `app/core/security.py` 和相关路由的角色枚举与前端映射一一对应，不存在遗漏角色导致鉴权异常的情况。

### 18. 安全问题修复

逐一检查并修复以下类别的安全问题：

**前端**：
- XSS：富文本/用户输入内容渲染前必须经过 DOMPurify 或等效库净化，禁止直接 `v-html` 渲染未经处理的用户输入。
- 敏感信息：localStorage 中除 token 外不存储明文用户信息（如密码、完整用户资料），token 过期后及时清除。
- 请求鉴权：所有需要登录的页面在路由守卫中校验 token 有效性（`meta.roles` 已有，补全漏项）。

**后端**：
- SQL 注入：确认所有数据库操作使用 SQLAlchemy ORM 或参数化查询，禁止字符串拼接 SQL。
- 越权访问：员工不能访问 `auditor`/`admin` 专属接口，后端路由依赖 `get_current_user` + 角色校验，检查并补全所有接口的角色守卫。
- 密码强度：注册和修改密码时后端校验密码长度不少于 6 位（与默认密码 `123456` 保持一致），可按需提升为 8 位。
- 文件上传：知识上传接口校验文件类型和大小限制，拒绝可执行文件上传。

## 第七部分：前后端对齐

### 19. 全面对齐前后端接口

- 梳理所有前端 API 调用（`src/api/` 目录下所有文件），与后端实际路由（`app/api/` 目录）逐一核对：
  - 路径是否一致（注意 `/api/` 前缀）
  - 请求方法（GET/POST/PUT/DELETE）是否匹配
  - 请求体/响应体字段名、类型是否一致
  - 分页参数格式是否统一
- 对存在不一致的接口，优先修改前端 API 层以对齐后端，避免改动后端接口影响其他功能。
- 如后端缺少本次新增功能所需接口，需在后端同步实现并在前端完成对接。

● 我已经分析清楚了两个问题的根因。下面先简要说明：

  问题 1：引用与问题不搭边 + 无法预览

  两个独立 bug：

  1. 无法预览原文 — src/views/search/Pc.vue:392 和 Mobile.vue:274 的 router-link
  跳转用的是 /kb/preview?id=xxx（query 参数），但路由定义的是
  /kb/preview/:docId（path 参数），永远命中 catch-all 被打回 /search。
  2. snippet 与提问不搭边 — backend/app/api/chat.py:75 把 chunk[0:150] 当
  snippet 返回。Chroma 命中的是整个 chunk（500 字），切片前 150 
  字很可能是首段无关的过渡文字，并未包含命中关键词，让用户感觉「不搭边」。

  问题 2：图谱切换案例/手册时节点不消失

  根因： 后端 backend/app/api/kg.py 抽取出的节点 type 只可能是 device | part | 
  fault | method 四种实体类型，根本没有 case 或 manual 类型的节点。前端
  src/views/kg/Pc.vue:83-87 的过滤逻辑写的是「n.type !== 'case' && n.type !== 
  'manual' → 直接放行」，所以切「案例/手册」永远不过滤任何节点。

  正确做法是依据节点的 source_docs[].doc_type 来判别归属：
  - 来源中存在 experience（经验分享） → 算「案例」
  - 来源中存在 pdf/docx/txt/md → 算「手册」

  后端 source_docs 已经携带 doc_type 字段了，所以这是纯前端修复。

● 全部修改已落地。下面是这两个问题的完整诊断与修复总结：

  ---
  问题 1：AI 引用与提问"不搭边"且无法预览

  根因

  1. 跳转 404 后被静默打回首页
  src/views/search/Pc.vue:392、Mobile.vue:274 的 router-link 用了 { path: 
  '/kb/preview', query: { id: h.docId } }，但路由定义是 /kb/preview/:docId（path
  参数，src/router/index.ts:27）。带 query 的形式永远命中 /:pathMatch(.*)*
  catch-all，被重定向回 /search，用户看起来"点了没反应"。
  2. snippet 与问题不相关
  backend/app/api/chat.py 旧实现是 (h["content"] or "")[:150]，取 chunk 的前 150
  字。但 Chroma 的命中颗粒是整个 500 字 chunk，命中关键词常出现在 chunk
  中段甚至末尾，前 150
  字往往是无关过渡段落，所以折叠面板显示的内容与提问看似无关。

  修复

  - Pc.vue / Mobile.vue：router-link :to 改为模板字符串
  \/kb/preview/${h.docId}``，命中正确路由。
  - backend/app/api/chat.py：新增
  _extract_keywords（提取问题中的中文≥2字片段和英文/数字 token，过滤停用词）+
  _build_snippet（以首个命中关键词为中心截取上下文窗口，前后加省略号；无命中才退
  回首段）。/api/chat/query 改用 _build_snippet。

  ---
  问题 2：知识图谱切"案例 / 手册"时节点不消失

  根因

  后端 backend/app/api/kg.py 用关键词正则抽取实体，节点的 type 字段只可能是
  device | part | fault | method 四种 —— 从未生成 type='case' 或 type='manual' 
  的节点。前端 src/views/kg/Pc.vue 与 Mobile.vue 的过滤逻辑里写 if (n.type !== 
  'case' && n.type !== 'manual') return true，所以"案例 /
  手册"按钮永远直接放行所有实体节点 —— 切换毫无效果。

  正确归属信息其实已经在 n.source_docs[].doc_type 字段里（后端 kg.py:281
  写入）：
  - experience → 案例（员工经验分享）
  - pdf / docx / txt / md → 手册

  修复

  - Pc.vue / Mobile.vue：filteredNodes / filtered 改为依据
  n.source_docs[].doc_type 判断。切"案例"时保留至少有一条 experience
  来源的节点，切"手册"时保留至少有一条非 experience 来源的节点，无 source_docs
  的节点在两种情况下都被屏蔽。

  - python -c "ast.parse(...)" —— 后端 chat.py 语法 OK。

● typecheck 0 错误。下面给出完整诊断和修复说明：

  根因

  这个 bug 是 localStorage 状态污染，不是后端 bug。

  现象 1：新创建的工单"默认全部完成"

  src/stores/workflow.ts 旧版用了一个没有任何隔离的本地存储键：
  const STEP_KEY = (orderId: string) => `workorder:steps:${orderId}`
  orderId 形如 t-5，不带用户 id、不带数据库 epoch。setFlow 时直接 stepDone.value
  = { ...persistedDone }，把这个键读进来覆盖前端状态。

  而 src/api/workflow.ts 的 ticketToFlow 又完全丢弃了后端返回的 
  t.progress，所以前端无法用真实进度校正本地缓存。

  典型触发场景：
  1. 之前用户在某个工单（比如 t-5）上把所有步骤勾完 → localStorage 留下
  workorder:steps:t-5 = {s1:true, s2:true, …}。
  2. 开发时清/重建过 loongchip.db，ticket id 从 1 重新自增。
  3. 任何用户新建工单撞到旧 id（比如又一个 t-5），进详情页时 store 从
  localStorage 读出旧的 {s1:true,…}，前端立刻显示全部勾完。
  4. 用户在步骤上稍一交互，watch(stepDone) 触发
  syncStepProgress，把这一份"假完成"的 [s1, s2, ...]
  推送给后端，污染就持久化了。

  现象 2："被自动添加到所有用户的作业指引中"

  后端 /api/ticket 把其他用户创建的工单放进 recommended
  数组返回给所有人（工单本就是平台级共享资源，README 写明），UI
  上叫"推荐工单"。但配合现象 1
  的视觉结果——别人那边也显示步骤已勾完——看起来就像"自动添加并完成"。

  实际上：推荐工单只是展示在「推荐工单」分组，用户没点"添加到我的工单"就不会进
  mine。这部分逻辑是对的。

  修复

  1. 后端进度变成唯一权威源（src/api/workflow.ts）

  - SopFlow 接口新增 stepDone?: string[] 和 status?。
  - ticketToFlow 把 t.progress.stepDone 透传到 SopFlow，不再丢弃。

  2. Store 改为以后端为权威（src/stores/workflow.ts）

  - 删除 STEP_KEY 和对 stepDone 的 watch+persist，永远不再把 stepDone 写 
  localStorage。
  - setFlow 用 new Set(f.stepDone) 直接重建 stepDone，完全不读 localStorage。
  - 子步骤 subDone

  与后端一致。

  workorder:checks:<userId>:t-N / workorder:subs:<userId>:t-N，加入用户隔离。

  3. 启动时清理历史污染键（src/stores/workflow.ts）

  根因

  后端 /api/ticket/{id}/timeline 早就有按角色分支：
  - _is_auditor(role) 命中 → 返回 {"grouped": [...]}（按用户分组所有事件）
  - 否则返回 {"events": [...]}（仅当前用户）

  _is_auditor 包含 auditor / leader / admin，所以管理员应该走分组分支。前端
  TicketTimeline.vue 模板原本是
  v-else-if="grouped.length"，只有数组非空才走分组分支，否则回退到 events 渲染。

  会导致"管理员只看到自己"的常见场景：
  1. 该工单仅 admin 自己产生过事件（自己创建、自己加入）—— grouped 长度为 1（仅
  admin 这一个用户分组），渲染上确实只能看到一个用户，用户感觉"和员工视角一样"。
  2. 若 grouped 为空但 events 非空（兼容旧后端或异常响应），前端会回退到员工视角
  的扁平列表，看起来就是"只看到自己的"。
  3. UI 没有任何标识表明当前是"审查视角"还是"员工视角"，管理员看到只有自己时无法
  分辨究竟是后端没返回分组、还是这条工单确实只有 admin 操作过。

  修复

  1. 后端 — 响应附带 viewer 标识与 creator（backend/app/api/ticket.py）

  让前端不靠"猜字段名"判断视角。

  - 兼容旧后端：审查员视角下若 grouped 缺失但 events 非空，把 events

#### `backend/app/core/security.py`

- `get_current_user` 解码后查询用户当前 `token_version`，与 `payload.tv` 严格比对；不一致抛 `HTTPException(401, "账号已在其他设备登录，请重新登录")`；旧 token（无 `tv`）兜底视为版本 1。

```python
def create_token(sub: str, role: str, token_version: int = 1) -> str:
    """FIX6 第 10 项：payload 中嵌入 token_version，配合 User.token_version 实现单点登录。"""
    payload = {
        "sub": sub, "role": role, "tv": token_version,
        "exp": datetime.utcnow() + timedelta(minutes=settings.JWT_EXPIRE_MIN),
    }
    return jwt.encode(payload, settings.JWT_SECRET, algorithm=settings.JWT_ALG)


def get_current_user(...):
    # ... 原有 sub / 用户存在校验 ...
    token_tv = payload.get("tv")
    current_tv = getattr(user, "token_version", 1) or 1
    if token_tv is None:
        token_tv = 1
    if int(token_tv) != int(current_tv):
        raise HTTPException(401, "账号已在其他设备登录，请重新登录")
    return user
```

#### `backend/app/api/auth.py`
- `register`：首次创建 `User(token_version=1)`，签发携带 tv=1 的 token。
- `login`：成功后 `u.token_version += 1` 并 `db.commit()`，再签发携带新 tv 的 token；其他设备旧 token 立刻失效。
- `change_password`：改密成功后同样 `+= 1`，强制其他设备重新登录。

### 前端
- `src/api/request.ts` 响应拦截器：识别 401 响应 `detail` 含 "其他设备" 字样时，通过 vant `showFailToast` 弹出专属提示，再跳转登录。

## 第 11 项：普通用户改密覆盖管理员密码

### 根因
单一 `app:token` localStorage key 被两个 Tab 共享：后登录者的 token 覆盖先登录者；后端 `change-password` 依赖请求头取出 `current_user`，因此普通用户改密时使用的是被覆盖后的管理员 token，落库到管理员 hash。

### 前端 — Token 存储隔离（根治）

#### `src/api/request.ts`
- 旧 `TOKEN_KEY = 'app:token'` 保留为 `LEGACY_TOKEN_KEY`，仅用于一次性迁移。
- 新增 `USER_TOKEN_KEY = 'user_token'` 与 `ADMIN_TOKEN_KEY = 'admin_token'`。
- 新增工具函数：
  - `isAdminContext()` —— hash 模式下读 `location.hash`，前缀 `#/admin` 视为管理端。
  - `activeTokenKey()` —— 返回当前上下文应使用的 key。
  - `readActiveToken()` —— 读取并按需迁移 `app:token` → `user_token`。
  - `writeActiveToken(t)` / `clearActiveToken()`。
- 请求拦截器从 `readActiveToken()` 取 token；401 时 `clearActiveToken()` 只清当前端的 key，**不影响另一端**。
- 兼容旧符号 `TOKEN_STORAGE_KEY`（指向 `USER_TOKEN_KEY`），同时导出 `USER_TOKEN_KEY / ADMIN_TOKEN_KEY / LEGACY_TOKEN_KEY` 备查。

#### `src/stores/user.ts`
- 不再 `import { storage } from '@/utils/storage'` + `TOKEN_STORAGE_KEY`；改用 `readActiveToken / writeActiveToken / clearActiveToken`。
- `login()`：`writeActiveToken(res.token)`。
- `hydrate()` token 失效与 `logout()`：`clearActiveToken()` 只清当前上下文（杜绝意外清空另一端）。

> **效果**：同浏览器同时打开用户端（普通路由）与管理端（`/admin/*`）时，两端读写不同 localStorage key，请求拦截器按 `location.hash` 决定附带哪一个 token，不会再发生 token 串扰。

### 后端 — 加固

#### `backend/app/api/auth.py` · `PUT /api/auth/change-password`
- **必须校验旧密码**：`if not verify_password(body.old_password, user.password_hash): raise 400 "原密码不正确"`。
- **显式按 id 锁定目标**：`target = db.query(User).filter(User.id == user.id).first()`，写入 `target.password_hash`，避免任何依赖混淆。
- 同步 `target.token_version += 1`（接 FIX6 第 10 项），强制其他端重新登录。
- 写入审计日志：`logger.info("[change-password] user_id=%s username=%s", target.id, target.username)`。

#### `backend/app/api/admin.py` · `PUT /api/admin/users/{id}/reset-password`
核对后已符合规范：
- 用路径参数 `user_id` 查询 `target_user = db.query(User).get(user_id)`，与 `current_user` 完全解耦。
- 写入 `u.password_hash`（即目标用户）。
- `require_admin` 依赖确保只能管理员调用；默认 admin 由 `_is_protected_admin` 在角色/用户名变更、删除处保护（此接口本身允许重置自身密码以便忘密时使用——但生产建议改用个人中心，超出本次修复范围）。

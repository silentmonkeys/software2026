为什么检索不到 + 描述不准
链路有 5 个环节出问题，按影响排序：

1. VL prompt 把方向带歪了（描述不准的直接原因）
llm.py:24 里 vl_describe() 的 prompt 写死成 "请简要描述图片中的设备状态与可能故障"，这是给"现场设备照片"准备的。你上传的是维修手册截图（一页文档），模型被这个 prompt 引导去找"设备故障状态"，自然不会去识别页面里的文字、章节标题、零件名词——所以描述就跑偏了。

2. 查询拼接后语义被稀释（检索不到的核心原因）
search.ts:75 当你不输入文字时，前端把 question 默认成 "请描述设备图片中的故障"。
rag.py:65 后端把它拼成：


请描述设备图片中的故障
（图片观察：图片显示一本摩托车维修手册的某一页，内容是…）
整段拿去 text-embedding-v3 做向量，前半句对知识库毫无语义信号，后半句又是"图片长什么样"而不是"图片里讲什么"，embedding 漂在一个跟手册具体内容很远的位置，自然检索不到对应文档。

3. PDF 入库时图片内容被完全丢弃（结构性问题）
parser.py:5-7 read_pdf() 只用 pypdf.extract_text() 抽文字流。这意味着：

摩托车手册里的图表、电路图、拆解图完全没进向量库
如果手册是扫描版 PDF（每页是图片），extract_text() 直接返回空字符串，整本书在 Chroma 里零向量——这种情况无论你怎么搜都搜不到
没有 OCR 流程（grep 不到 tesseract/pytesseract）
4. 检索没有相似度阈值（放大问题）
rag.py:41-52 search() 无条件返回 Top-5，不看 distance 是否够低。即使你的查询跟所有 chunk 都很远，依然返回 5 条最接近的——给你"搜到了但全不相关"的假象，掩盖了实际是检索失败。

5. VL 输出非结构化、无 OCR
llm.py:24-41 Qwen-VL 输出自由文本，没要求它做 OCR / 转写文字 / 提关键术语。手册截图里大量的关键词（型号、零件名、步骤号）都没被提取到查询里。

怎么修（按性价比排）
改 VL prompt + 多模态分支（最快，几行代码）：检测如果用户没输入 question，把 prompt 改成 "请识别图片中的文字内容、关键术语、设备/零件名称、章节标题，按要点列出"，把输出当查询主体而不是附属。
前端不要兜底成 "请描述设备图片中的故障"：要么真用空 question + VL 输出当查询，要么提示用户必须输入关键词。
加相似度阈值：distance > 0.5（或 cosine < 0.3）就当没搜到，前端显示 EmptyState，别强行返回 Top-5。
PDF 入库增加 OCR：扫描版手册必须过 OCR（pytesseract 或调 Qwen-VL 逐页识别），否则你库里根本没数据。
可选：入库阶段对每页图表也跑一次 VL 描述，把描述 chunk 一起入向量库。
